# Coding Standards: pico-hubasic

このプロジェクトでは、保守性が高く、バグの混入しにくい高品質な組み込みコードを目指します。

## 1. 開発プロセス (TDD & Test)
- **TDD (Test Driven Development) の徹底**:
    1. 機能を実装する前に、期待される動作を定義するテストコードを書く。
    2. テストを実行し、失敗（Red）を確認する。
    3. テストをパスするための最小限の実装を書く（Green）。
    4. コードをクリーンに整理する（Refactor）。
- **単体テストのカバレッジ**:
    - **全モジュールでカバレッジ 80% 以上**を維持してください。
    - 特に、数式パース、トークン変換、メモリ管理などのロジック部分は 90% 以上を目指します。
- **テストの実行**:
    - コードを変更した際は、必ず既存の全テストスイートを実行し、リグレッション（先祖返り）がないか確認してください。

## 2. 関数・構造の設計 (可動性の向上)
- **浅いネスト**: 
    - `if`, `for`, `while` などのネストは原則 **3階層まで** とします。
    - ガード節（早期 return）を積極的に活用し、正常系の処理をインデントの浅い場所に記述してください。
- **関数のサイズ**:
    - 1つの関数は、**1画面（約40〜60行）以内**に収めるようにしてください。
    - 1つの関数が担う責務は1つだけに限定（単一責任の原則）してください。
- **循環的複雑度 (Cyclomatic Complexity)**:
    - 複雑な条件分岐は避け、判定ロジックを小さな関数に切り出してください。

## 3. 組み込み向け実装ルール
- **動的メモリ確保の制限**:
    - 実行時（メインループ内）での `new`, `malloc`, `std::vector::push_back` 等の動的確保は禁止します。
    - 原則として、起動時に必要なバッファを静的に確保するか、固定長配列を使用してください。
- **浮動小数点の扱い**:
    - 計算精度が重要な部分は、意図せず `double` から `float` へ暗黙の型変換が行われないよう注意してください。
- **マジックナンバーの排除**:
    - ピン番号、レジスタ値、BASICのトークン値などは、直接数値を書かず `constexpr` または `enum class` で定義してください。
- **揮発性変数の明示**:
    - 割り込みハンドラとメインループの両方で参照する変数は `volatile` を付与し、必要に応じて排他制御を行ってください。

## 4. 命名規則
- **変数・関数**: `snake_case` (例: `current_line_number`, `get_token()`)
- **クラス・型名**: `PascalCase` (例: `BasicInterpreter`, `MemoryManager`)
- **定数・マクロ**: `UPPER_SNAKE_CASE` (例: `MAX_HEAP_SIZE`, `GPIO_LCD_CS`)
- **メンバ変数**: 接尾辞にアンダースコア（例: `buffer_`, `is_running_`）を推奨します。

## 5. コメント
- **Doxygen形式**: 公開関数やクラスのヘッダには、引数と戻り値を明記した Doxygen 形式のコメントを記述してください。
- **Why を書く**: 「何をしているか」はコードから読み取れるため、複雑な処理には「なぜそうしているか（背景や制約）」を記述してください。

## 6. 継続的リファクタリング
- **ボーイスカウト・ルールの適用**: コードを変更する際は、その周辺の既存コードも「来た時よりも美しく」することを心がけてください。
- **リファクタリングのタイミング**: 
    - 重複したコード（Copy & Paste）を発見した時。
    - 1つの関数の責務が大きくなりすぎた時。
    - `coding-standards.md` の設計ルール（ネストの深さや行数）に抵触し始めた時。
- **安全なリファクタリング**: 
    - 必ず既存の単体テストがパスしている状態で開始し、リファクタリング後もテストがパスすることを確認してください。
- **提案**: 大きな構造変更が必要な場合は、実装前にリファクタリング案をユーザーに提示し、承認を得てください。
